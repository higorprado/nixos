# DMS Wallpaper Wiring Cleanup Plan

## Goal

Improve the readability and maintainability of [dms-wallpaper.nix](../../../modules/features/desktop/dms-wallpaper.nix) without changing the functional role of `dms-awww`.

This is **not** a redesign of the integration.

`dms-awww` is intentionally responsible for:
- intercepting the DMS wallpaper integration path,
- preserving preview/integration behavior,
- delegating wallpaper work to `swww`,
- triggering the theme update path used by `DankMaterialShell`.

The cleanup target is only the Nix wiring around that integration.

## Non-Goals

- changing the purpose or boundaries of `dms-awww`
- splitting wallpaper and theme behavior into separate features
- redesigning the `dms-awww` package itself
- changing the runtime contract of `dms.service`, `awww-daemon.service`, or `dms-awww.service`
- moving host-scoped policy into shared features

## Current Pain Points

1. Static `dms-awww/config.toml` is generated by `home.activation`.
   - This is noisy and harder to audit than a declarative HM config file.

2. `awww` wrapper definition is duplicated in both the NixOS and HM branches.
   - The module currently redefines the same binary more than once.

3. Package ownership is harder to read than necessary.
   - The module currently mixes `environment.systemPackages` and HM runtime packaging in a way that is harder to reason about than it needs to be.

4. The file is doing too much wiring inline.
   - It is still one valid functional unit, but the presentation can be cleaner.

## Target Shape

After cleanup, the module should read as:

1. shared local definitions
   - `awww`
   - `awww-daemon` if still needed as a wrapper
   - `dmsPackage`
   - `runDmsAwww`

2. declarative user config
   - `xdg.configFile."dms-awww/config.toml".text`
   - copy-once mutable DMS settings stays separate

3. user services
   - `awww-daemon`
   - `dms-awww`

4. only the minimum necessary system-side packaging

### Phase 1: Move Static Config to HM Declarative Config

Target:
- replace `provisionDmsAwwwConfig` activation snippet with declarative HM config

Change:
- remove the inline `cat > ~/.config/dms-awww/config.toml`
- replace with:
  - `xdg.configFile."dms-awww/config.toml".text = ...`

Keep:
- `provisionDmsSettings` as copy-once, because that file is intentionally mutable

Validation:

```bash
./scripts/run-validation-gates.sh structure
nix build .#nixosConfigurations.predator.config.home-manager.users.higorprado.home.path -o /tmp/dms-wallpaper-phase1-hm
nix store diff-closures /tmp/dms-wallpaper-before-hm /tmp/dms-wallpaper-phase1-hm
```

Expected:
- HM diff only where the config file provisioning mechanism changes
- no semantic regression in service definitions

Commit target:
- `refactor: make dms-awww config declarative`

### Phase 2: Deduplicate Wrapper Definitions

Target:
- define `awww` once in a shared local `let`

Change:
- remove duplicate local wrapper declarations
- keep a single canonical wrapper binding if the wrapper is still needed

Validation:

Expected:

Commit target:
- `refactor: dedupe dms wallpaper wrappers`

### Phase 3: Re-evaluate Package Ownership

Question:
- do `awww`, `awww-daemon`, and `dms-awww` really need to live in `environment.systemPackages`, or can the runtime be fully owned by HM?

Decision rule:
- if a package exists only to support user services or user-facing commands in this integration, prefer HM ownership
- keep system-side ownership only if the package is intentionally part of the host-wide system environment

Change:
- move package ownership to the cleanest side without changing runtime behavior

Validation:

```bash
./scripts/run-validation-gates.sh structure
./scripts/run-validation-gates.sh predator
nix build .#nixosConfigurations.predator.config.system.build.toplevel -o /tmp/dms-wallpaper-phase3-system
nix build .#nixosConfigurations.predator.config.home-manager.users.higorprado.home.path -o /tmp/dms-wallpaper-phase3-hm
nix store diff-closures /tmp/dms-wallpaper-phase2-system /tmp/dms-wallpaper-phase3-system
nix store diff-closures /tmp/dms-wallpaper-phase2-hm /tmp/dms-wallpaper-phase3-hm
```

Runtime check after `nh os test` or `switch` on `predator`:

```bash
systemctl --user status dms.service --no-pager
systemctl --user status awww-daemon.service --no-pager
systemctl --user status dms-awww.service --no-pager
```

Commit target:
- `refactor: simplify dms wallpaper package ownership`

```bash
nix build .#nixosConfigurations.predator.config.system.build.toplevel -o /tmp/dms-wallpaper-after-system
```

Success criteria:
- no regression in DMS startup
- no regression in `dms-awww`
- file is easier to read
- static config is no longer provisioned by activation script
- wrapper/package wiring is less duplicated

## Risks

1. HM config file replacement may accidentally change mutability expectations.
   - Mitigation: only convert `dms-awww/config.toml`; keep `settings.json` copy-once.

2. Package ownership move may remove commands from a context where the service still expects them.
   - Mitigation: validate evaluated service config and runtime after test switch.

3. Over-cleanup may erase the intentional functional coupling of wallpaper + theme update.
   - Mitigation: preserve `dms-awww` as one functional integration unit.

## Notes for the Agent

- Do not “fix” the module by splitting the functional responsibilities of `dms-awww`.
- The right target is cleaner expression of the same integration.
- Prefer small slices and empty closure diffs where possible.
